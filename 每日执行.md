> 用途：让模型每天 1 句话就能进入状态，标准化执行"拉数据 → 给推荐 → 结算昨日 → 同步站点 → 提交推送"全流程。

- **当前余额**：892 元（2026-06-09 09:48 截止）

  ```
  单子出票 / 确认 → -cost（一次性扣完）  ← 唯一扣钱点
  结算 won  → +payout（加奖金）
  结算 lost → 不动（cost 已扣过）
  结算 pending → 不动
  ```

  余额公式：`1000 − Σ(bet.totalCost) + Σ(bet.payout where result="won")`
- **资金分配**：60% 世界杯正赛 / 25% 热身赛 / 15% 应急
- **长期策略**：凯利 1/3 压缩、单日亏损 ≤25%、大名单轮换 ≥5 人自动降仓 50%

## 1. 每日 6 步执行流程（2026-06-15 调整）

> **关注范围仅限世界杯正赛**（`data/matches.json` 的 M001-M104）。其它赛事不拉、不推。
> **AI 模拟结算已移除**——用户后续单独触发调整；本流程只管"出推荐模拟单"，不管"结旧单"。
> **多 LLM 比分预测不在本流程**——`data/predictions.json` 由用户手动录入（Deepseek / Claude / MiniMax / Kimi / GPT-5 mini 等多模型结果），agent **不主动调用**、**不写入**。统计页只读消费该文件做命中复盘。

### 顺序速查

| Step | 任务                                | 子 agent？             |
| ---- | ----------------------------------- | ---------------------- |
| 1    | 拉完赛结果 + 更新比分               | ✅                     |
| 2    | 重算积分（FIFA Standings）          | ❌ 上下文内            |
| 3    | 统计（build_index + AI 命中统计）   | ❌ 上下文内            |
| 4    | 拉未开赛已放出赔率（仅世界杯正赛）  | ❌（>10 场才开 agent） |
| 5    | 执行 `npm run modeling:all` 拿预测  | ❌ 上下文内            |
| 6    | 基于赔率-结果模型推荐给出当日模拟单 | ❌ 上下文内            |

### Step 1 · 拉完赛结果 + 更新比分（子 agent）

**触发条件**：有正赛已开赛（>2 小时）且 `data/results/<mid>.json` 缺该场结果（且该 mid 属于 `data/matches.json` 的 M001-M104）。

> ⚠️ **范围红线**：mid 不在 `data/matches.json` 的完赛**不录**（国际赛 / 热身赛 / 其它友谊赛）。即便 `data/results/` 已有历史数据（6-12 之前的热身赛结果），也**不再新增**。

> 📐 **数据 schema（results）**：单一来源 = `data/results/<mid>.json`（per-mid 拆分，与 `matches_status.json.result_file` 对齐）。
> 前端用 Vite `import.meta.glob('../data/results/*.json', { eager: true })` 读，**不再用扁平的 `data/results.json`**（已删除）。
> 每场文件至少含：`matchId / homeScore / awayScore / halfTime / scorers / wentToPenalties / penaltyScore`，
> 其中 `halfTime` 和 `scorers` 是**必填**（`update-result.js` 脚本会校验）。

派一个 `general_purpose_task` 子 agent，子 agent 任务清单：

1. **正赛阶段**（6-12 起）数据源：
   - 比分 / 进球者 / 半场：[FIFA Match Centre](https://www.fifa.com/en/match-centre) — 官方权威，scorers 的 minute/type 直接抄这里
2. `node scripts/update-result.js <matchId> <homeScore> <awayScore> --half-time=h:a --scorer=team:player:minute:type [...]` 写入 `data/results/<matchId>.json`
   - 必传 `--half-time`（bqc 玩法结算需要半场比分）
   - 比分非 0:0 时 `--scorer` 至少 1 个（保证前端"进球者"区块有内容）
   - 走点球时加 `--penalties=h:a`
3. 子 agent 报告：今日新增了哪些 mid 的比分，**显式列出跳过的非正赛 mid**

---

### Step 2 · 重算积分（FIFA Standings）

> 只重算**正赛**小组的积分。其它赛事不动 `data/groups.json`。

1. 拉 [FIFA Standings · 世界杯 2026](https://www.fifa.com/en/tournaments/mens/worldcup/canadamexicousa2026/standings) 12 个小组当前积分（胜 3 / 平 1 / 负 0；净胜球 = GF - GA；总进球 GF / 总失球 GA）
2. 直接编辑 `data/groups.json`（12 个小组整组覆盖更新）
3. **校验**：每组 4 队 + 每队比赛场数 = 小组已完赛场数（如有错立即标红停手）

---

### Step 3 · 统计

1. `node scripts/build_index.js` 刷新 `data/matches_index.json`
2. 看 `js/pages/stats.js` 的 dashboard（AI 模型准确率榜）— 自动从 `predictions.json` + `results/*.json` 算
3. 复盘：哪些 AI 命中 / 哪些未中，新发现写进 `records/reflections.md`

> 📌 **AI 模拟结算不在本流程**（用户后续单独触发）。本步骤不修改 `data/bets.json` 的 pending → won/lost。
> 📌 **`data/predictions.json` 只读不写**：本步骤不调用 `add-prediction.js`，AI 模型命中统计是消费用户已手动录入的数据。

---

### Step 4 · 拉未开赛已放出的场次赔率（仅世界杯正赛）

> ⚠️ 范围红线：**只拉** `data/matches.json` 的 M001-M104 对应的场次赔率。其它赛事不抓。

用 chrome-devtools-mcp（**新开 isolated context 避免实例冲突**）：

1. 导航到 `https://www.sporttery.cn/jc/jsq/zqspf/`
2. 抓所有**在售** + **待开售**的**世界杯正赛**场次（基本/赔率/近期/伤停/赛程）
3. **过滤**：每个 mid 必须能映射回 `data/matches.json` 的 M-id（`scripts/scrape_match.js` 的 `MATCH_IDS` 列表已经做了这层映射）；**不映射的跳过**
4. 每场写到 `data/<今日>/<周X-NNN_主_vs_客>.json`（用 `Write` 工具，不要 cat）
5. **比分表定位**：动态按表头含 `"1:0"` 找表（避免让球调整后表格顺序变）
6. **待开售场次**：赔率全 `null`，但保留 mid/code/开售时间，建监控后续补抓
7. 重跑 `node scripts/build_index.js`

> 大批量重抓（>10 场）才开子 agent。

### Step 5 · 执行 modeling 拿模型预测

```bash
npm run modeling:all
```

等价于：

1. `modeling:prepare` — 从 `data/matches_status.json` + `data/results/*.json` 提炼
2. `modeling:train` — 训练 3 个模型（胜平负 / 让球盘路 / 比分 Poisson）
3. `modeling:predict` — 对未开赛的世界杯正赛出推荐

主入口产物：`modeling/artifacts/predict_unplayed.json` —— 18+ 条结构化推荐（含 `win.pick` + `win.confidence` + `handicap.verdict` + `score_top3`）。

> 📌 **modeling 训练样本仅含世界杯标签完赛场次**（11 场，截至 2026-06-15）。国际赛热身已剔除并落 `modeling/data/international_warmup.json` 留档，不进训练。

### Step 6 · 基于赔率-结果模型推荐出当日模拟单

> ⚠️ 写推荐前必先打开 `records/reflections.md` 通读一遍，确认当天的玩法/仓位/让球换算没触犯 R-00X。

**输入**：`modeling/artifacts/predict_unplayed.json`（模型给的胜平负 + 让球 + Top-3 比分）。

写到 `records/<今日>.md`，四段式：

```markdown

（数据源标注：modeling 模型推荐 + sporttery 实际赔率 / 资金分配 / 单关推荐表 / 可选串关 / **每场必出「全玩法预测」表**）

### 1.x 比赛名

**推荐玩法 + 仓位**：（例：让-1 主胜，80 元）

**理由 ≥ 4 条**（R-006 强制）：

1. **模型推荐**：modeling/artifacts/predict_unplayed.json 给出 pick=home / confidence=⭐⭐⭐，Top-3 比分 [2-0, 1-0, 2-1]
2. **让球换算**（rqspf 必含）：让-1 主胜 = 主队让 1 球后仍胜，实际需 2:0 / 3:1 / 4:2 …
3. **实力对比 / 近期状态 / 伤停**：近 3 场 A 队胜胜平、B 队负负平；主力 X 赛前 30 分钟复核
```

>
> ⚠️ **R-007 赔率市场情绪验证**（**2026-06-09 用户反馈后替代 5 模型陪审团**）：

>

> **5 模型陪审团已撤**（用户反馈"5 模型推测不可作为你的数据依赖"）。改用"**赔率市场 + modeling 模型**"作为独立信息源。

## 4. data/bets.json Schema（双形态）

### 单关（lines）

```json
{
  "id": "...",
  "type": "champion|match",
  "title": "...",
  "matchLabel": "...",
  "date": "ISO",
  "stakeMultiplier": 1,
  "lines": [
    {
      "pick": "ESP",
      "multiplier": 25,
      "odds": -4.75,
      "result": null,
      "note": null
    }
  ],
  "note": "..."
}
```

### 串关（picks，实票样式）

```json
{
  "id": "...",
  "type": "match",
  "title": "[模拟] 让球 串关（2×1 + 3×1）",
  "matchLabel": "...",
  "date": "ISO",
  "stakeMultiplier": 1,
  "parlayType": ["2x1", "3x1"],
  "picks": [
    {
      "match": "周一201",
      "home": "NED",
      "away": "UZB",
      "handicap": -2,
      "pick": "home",
      "pickLabel": "主胜",
      "odds": 2.69
    }
  ],
  "unitCost": 2,
  "combinations": 4,
  "totalCost": 8,
  "maxReturn": 83.67,
  "result": "pending",
  "note": "..."
}
```

**渲染分发**：`renderBet` 看是否有 `picks` → `renderParlayCard`（单卡实票样式） / `renderBetCard`（lines 展开）。

**串关套餐注数公式**（R-004 #6 · **不要手填 `combinations`**，按 `parlayType` 算）：

| 套餐写法      | picks 长度 | parlayType            | combinations 公式            | 注数 |
| ------------- | ---------- | --------------------- | ---------------------------- | ---- |
| `3场-2关`     | 3          | `["2x1"]`             | C(3,2)                       | 3    |
| `3场-2,3关`   | 3          | `["2x1", "3x1"]`      | C(3,2) + C(3,3) = 3+1        | 4    |
| `3场-3关`     | 3          | `["3x1"]`             | C(3,3)                       | 1    |
| `4场-2,3,4关` | 4          | `["2x1","3x1","4x1"]` | C(4,2)+C(4,3)+C(4,4) = 6+4+1 | 11   |

出票时如发现 `combinations` 与 `parlayType` 算出来对不上，**先停下**算清楚再写。
6-8 那张 `rqspf-parlay-2026-06-09` 票就是 `3场-2,3关` = 4 注 × 2 元 = 8 元。

## 7. 已知坑（预防性）

- **Chrome MCP 实例冲突**：`The browser is already running for ...` → 新页面用 `isolatedContext: "wc2026-ai-<日期>"`
- **PowerShell 控制台 buffer 报错**：长命令用 `Out-String -Stream` 或拆短
- **Node ESM 报错** `require is not defined` → 项目 `package.json` 已 `"type": "module"`，脚本统一用 `import`
- **比分表定位失败**：让球+/- 调整后 `tables[3]` 不一定是比分表，必须按表头 `"1:0"` 动态找
- **待开售场次没赔率**：spf/rqspf/bf 全 `null` 正常，售出后再补抓（写入 `rqspf_history` 第一条）
- **未在 teams.json 的球队**（如 PER/NIR）：先 `data/teams.json` 加条目再渲染，否则会 fallback 成 `?` 占位
